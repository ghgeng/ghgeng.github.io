策略模式

# 策略模式

策略模式建议找出负责用许多不同方式完成特定任务的类， 然后将其中的算法抽取到一组被称为策略的独立类中。

名为上下文的原始类必须包含一个成员变量来存储对于每种策略的引用。 上下文并不执行任务， 而是将工作委派给已连接的策略对象。

上下文不负责选择符合任务需要的算法——客户端会将所需策略传递给上下文。 实际上， 上下文并不十分了解策略， 它会通过同样的通用接口与所有策略进行交互， 而该接口只需暴露一个方法来触发所选策略中封装的算法即可。

因此， 上下文可独立于具体策略。 这样你就可在不修改上下文代码或其他策略的情况下添加新算法或修改已有算法了。

  • 上下文 （Context） 维护指向具体策略的引用， 且仅通过策略接口与该对象进行交流。

    就是上下文类中有一个顶级接口，具体实现是有子类决定的

  • 策略 （Strategy） 接口是所有具体策略的通用接口， 它声明了一个上下文用于执行策略的方法。

    就是一个接口中定义的成员方法

  • 具体策略 （Concrete Strategies） 实现了上下文所用算法的各种不同变体。

    就是子类实现接口的方法，执行自己的业务逻辑

  • 当上下文需要运行算法时， 它会在其已连接的策略对象上调用执行方法。 上下文不清楚其所涉及的策略类型与算法的执行方式。

    子类实现方法

  • 客户端 （Client） 会创建一个特定策略对象并将其传递给上下文。 上下文则会提供一个设置器以便客户端在运行时替换相关联的策略。

    就是new一个子类，set到上下文中，然后调用子类方法

# 示例一

  • 计算类策略接口，每个数字都可以加减乘除运算
public interface CalculateStrategy {
    public Integer execute(Integer a, Integer b);
}

@Data
public class Context {
    // 上下文会维护指向某个策略对象的引用。上下文不知晓策略的具体类。上下
    // 文必须通过策略接口来与所有策略进行交互。
    private CalculateStrategy strategy;
    // 上下文会将一些工作委派给策略对象，而不是自行实现不同版本的算法。
    public Integer executeStrategy(int a, int b) {
        return strategy.execute(a, b);
    }
}

# 示例二

  • 每个车有不同的行为，都可以加油和鸣笛，但是速度不一样有快有慢的，可以把相同的行为放在一起，不同行为各自生成自己的策略。
public interface CarBehavior {

    // 车可以加油
    public void cheer();

    // 车可以鸣笛
    public void whistle();

}

  • 把车的速度提取一个策略
@Data
public abstract class Car implements CarBehavior {
    private String name;

    // 每个车跑的有快有慢
    public abstract void run();

    public void detail() {
        run();
        cheer();
        whistle();
    }

    @Override
    public void cheer() {
        System.out.println(this.name + "加油~~");
    }

    @Override
    public void whistle() {
        System.out.println(this.name + "鸣笛~~");
    }
}
