# maven生命周期

# 1.mvn clean

mvn clean 指令执行时，Maven 会查找项目的 pom.xml 文件中定义的清理生命周期阶段。
在这个生命周期阶段中，Maven 默认绑定了 clean:clean 插件目标，该目标负责清理项目构建生成的所有文件。
这些文件通常包括 target 目录下的所有内容，但也可能包括其他在 pom.xml 中通过配置指定的文件和目录。

命令解释

  • 清理构建产物：删除之前构建生成的所有文件，包括编译后的 .class 文件、打包后的 JAR 或 WAR 文件、测试报告等。这样做可以确保下一次构建是基于最新的源代码和配置进行的，避免了构建产物之间的潜在冲突或干扰。
  • 准备新的构建：在进行新的构建之前，执行 mvn clean 可以确保工作区是干净的，从而避免因为遗留的构建产物导致的构建失败或不一致的问题。
  • 节省磁盘空间：随着时间的推移，构建产物可能会占用大量的磁盘空间。定期执行 mvn clean 可以帮助释放这些空间，让开发环境更加整洁和高效。

注意

mvn clean 只会删除构建过程中生成的文件，而不会删除源代码、配置文件或其他不属于构建产物的文件。此外，执行 mvn clean 后，如果想要重新构建项目，需要再次执行 mvn 命令（如 mvn compile、mvn package 等）来生成构建产物。

# 3.mvn compile

提示

mvn compile命令只会执行到编译阶段，不会执行后续的测试、打包等阶段。如果你想要执行更完整的构建过程，可以使用 mvn install（编译、测试、打包，并将打包的构件安装到本地仓库）或 mvn package（编译、测试、打包）等命令。

  • 解析 POM：Maven 首先会解析项目的 pom.xml 文件，这个文件是 Maven 项目的核心配置文件，包含了项目的配置信息，如项目依赖、插件配置等。
  • 依赖解析：Maven 会根据 pom.xml 中定义的依赖关系，从本地仓库或远程仓库下载并解析项目所需的依赖库。如果本地仓库中不存在所需的依赖库，Maven 会尝试从远程仓库下载它们。
  • 编译源代码：在依赖库准备就绪后，Maven 会执行编译过程。默认情况下，Maven 使用 Java 编译器（javac）来编译项目的.java源代码文件，生成.class字节码文件。这些字节码文件会被放置在项目的target/classes目录下（这是 Maven 的默认构建目录结构）。
  • 处理资源文件：虽然mvn compile命令主要关注源代码的编译，但Maven也会在这个过程中处理资源文件（如配置文件）。这些资源文件会被复制到target/classes目录下，以便在打包或运行时使用。

注意

  • Maven 的构建过程是基于生命周期的，每个生命周期阶段都会触发一系列插件目标。
  • 在 compile 阶段，Maven 会调用 maven-compiler-plugin 的 compile 目标来执行编译操作。
  • 这个插件是 Maven 的核心插件之一，用于处理 Java 源代码的编译。

# 5.mvn package

提示

mvn package 是 Maven 命令行中的一个指令，用于编译项目的源代码、处理资源文件，并执行项目的打包过程。打包过程会将编译后的代码以及所需的资源、依赖等打包成一个可分发的格式，如 JAR（Java 归档文件）、WAR（Web 应用归档文件）等，具体取决于项目的 pom.xml 文件中配置的打包类型。

命令解释

  • 编译源代码：首先，Maven 会查找项目的 pom.xml 文件中定义的构建生命周期阶段，并执行直到 compile 阶段（如果还没有执行的话）。compile 阶段会编译项目的源代码，生成.class文件。
  • 处理资源文件：在编译源代码之后，Maven 会处理资源文件（如配置文件），将它们复制到指定的目标目录中。
  • 打包：接下来，Maven 会执行 package 生命周期阶段，并触发 package 插件目标。这个阶段会根据项目的打包类型（在pom.xml的<packaging>元素中指定）将编译后的代码、资源文件和依赖库打包成一个可分发的归档文件。

# pl

mvn -pl 是 Maven 命令行工具的一个选项，用于指定要构建的项目列表（Project List）。在这个上下文中，-pl 或 --projects 选项后面跟着的是要构建的一个或多个项目的标识符（通常是项目的目录名或者模块名），这些标识符用于 Maven 的多模块项目中。
Maven 支持构建包含多个模块的项目，这些模块通常位于项目的根目录下的不同子目录中。每个模块都是一个独立的 Maven 项目，具有自己的 pom.xml 文件。但是，有时你可能只想构建这些模块中的一部分，而不是全部。这就是 -pl 或 --projects 选项的用武之地。

  • 使用示例 :

假设你有一个多模块项目，项目结构如下：

MyProject/  
|-- pom.xml  
|-- moduleA/  
|   |-- pom.xml  
|-- moduleB/  
    |-- pom.xml
1
2
3
4
5
6

如果你想构建 moduleA 和 moduleB，你通常会在 MyProject 目录（包含顶级 pom.xml 的目录）中运行 Maven 命令。但是，如果你只想构建 moduleA，你可以使用 -pl 选项：

# 这条命令会告诉 Maven 只构建 moduleA，忽略 moduleB 和其他任何未被显式指定的模块。

mvn clean install -pl moduleA
1
2
3

注意

使用-pl时，指定的模块名或目录名必须是相对于 Maven 项目根目录的。
-pl选项经常与-am（--also-make）或-amd（--also-make-dependents）选项结合使用，以确保构建所选模块及其依赖的模块。
在大型多模块项目中，合理使用-pl可以显著提高构建速度，因为你可以只构建你当前工作所需的部分。
